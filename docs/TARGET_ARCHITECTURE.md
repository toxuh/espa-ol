# Целевая архитектура

## Архитектурная цель

Приложение остаётся небольшим монолитом, но получает нормальные границы:

```text
Browser
  |
  | HTTPS / home network
  v
Existing Traefik on Synology
  |
  v
Next.js container
  |-- App Router pages and layouts
  |-- small interactive Client Components
  |-- Server Actions / Route Handlers
  |-- domain services: daily plan, checking, stats, streak
  |
  v
PostgreSQL container
```

Отдельный backend-сервис для этого проекта не нужен. Next.js обслуживает интерфейс и внутренние HTTP-операции, Prisma отвечает за доступ к PostgreSQL.

## Технологический контракт

| Область | Решение |
|---|---|
| Framework | Next.js, App Router, TypeScript |
| UI | Tailwind CSS + shadcn/ui |
| База | PostgreSQL |
| ORM | Prisma |
| Валидация | Zod на границах ввода |
| Контейнеризация | Docker Compose |
| Reverse proxy | существующий Traefik, не входит в compose проекта |
| Хостинг | Synology DS923+ |
| Профиль | выбор имени, идентификатор в `localStorage` |
| Тесты | unit для доменной логики, integration для БД, browser E2E для основных сценариев |

Конкретные версии зависимостей фиксируются при scaffold по актуальным стабильным релизам. В production должны использоваться только поддерживаемые и исправленные версии Next.js, React, Prisma и PostgreSQL.

## Границы слоёв

### Presentation

- маршруты и layouts App Router;
- Server Components для загрузки страниц;
- минимальные Client Components для форм, выбора варианта, `localStorage` и мгновенной обратной связи;
- компоненты shadcn/ui используются как исходные UI-примитивы;
- текущая визуальная структура и тексты сохраняются настолько близко, насколько разумно.

### Application

Сценарии приложения:

- выбрать профиль;
- пройти placement-тест;
- вручную изменить уровень;
- получить или создать день;
- отправить ответ;
- отметить чтение или аудирование;
- показать или оценить перевод;
- получить анализ дня, историю и статистику.

### Domain

Чистые функции без React и Prisma:

- нормализация ответа;
- проверка каждого типа задания;
- расчёт placement-уровня;
- выбор допустимых CEFR-уровней;
- ранжирование кандидатов;
- генерация дневного плана;
- проверка полноты дня;
- расчёт streak;
- агрегация статистики.

Эти функции должны принимать явные данные и управляемый источник случайности. В тестах используется детерминированный seed или подставной random-функционал.

### Persistence

- Prisma schema;
- репозитории или узкие query-функции;
- транзакции для первой попытки, статистики и завершения дня;
- seed учебного контента;
- миграции схемы.

UI не должен выполнять Prisma-запросы напрямую из Client Components.

## Маршруты интерфейса

Предлагаемый минимальный набор:

| Маршрут | Назначение |
|---|---|
| `/` | выбор/восстановление локального профиля и переход в приложение |
| `/placement` | выбор уровня или тест |
| `/practice` | практика текущего дня |
| `/theory` | теория и закрепление текущего дня |
| `/history` | календарь и выбор дня |
| `/history/[date]` | сводка конкретного дня |
| `/progress` | накопительная статистика и смена уровня |

Маршруты могут быть сгруппированы в `src/app/(app)`, но URL должен оставаться простым.

## Идентификация профиля

### Первая версия

1. На `/` браузер читает `espanolDiario.profileId` из `localStorage`.
2. Если идентификатора нет или профиль удалён, показывается список семейных профилей и ввод имени.
3. После выбора в `localStorage` сохраняются как минимум `profileId` и отображаемое имя.
4. Client Component передаёт `profileId` во внутренние запросы.
5. Сервер всегда проверяет существование активного профиля и использует его `id` во всех запросах.
6. В интерфейсе постоянно видно, чей профиль открыт, и есть действие «Сменить профиль».

Рекомендуется создавать профили из заранее заданного семейного списка или отдельной простой служебной команды, чтобы случайные опечатки не плодили пользователей. Если будет выбран свободный auto-create по имени, это нужно явно закрепить в `OPEN_DECISIONS.md`.

### Граница безопасности

Это не аутентификация. Заголовок или `profileId` можно подменить. Ограничение доступа обеспечивается домашней сетью, VPN или Traefik. Если приложение станет доступно из публичного интернета, перед публикацией обязательна настоящая серверная сессия.

## Серверный API

Можно использовать Server Actions для простых мутаций и Route Handlers для клиентских запросов, которым нужен `profileId` из `localStorage`. Предлагаемый внутренний контракт:

- `GET /api/profiles`;
- `POST /api/profiles/select` или клиентский выбор без серверной мутации;
- `POST /api/placement/complete`;
- `GET /api/day?date=YYYY-MM-DD`;
- `POST /api/day/grammar-answer`;
- `POST /api/day/vocab-answer`;
- `POST /api/day/conjugation-answer`;
- `POST /api/day/translation-from`;
- `POST /api/day/translation-to`;
- `POST /api/day/reading`;
- `POST /api/day/listening`;
- `GET /api/history`;
- `GET /api/progress`.

Это внутренняя граница, а не обещание публичного API. Точные endpoints допустимо объединить, если сценарии и транзакционные гарантии сохранятся.

## Создание дня

Создание должно быть атомарным и идемпотентным:

1. Сервер определяет дату в `APP_TIMEZONE`, по умолчанию `Europe/Madrid`.
2. Ищет `DailyPlan` по `(profileId, localDate)`.
3. Если день существует, возвращает его без регенерации.
4. Если нет, внутри транзакции получает статистику и недавнюю историю, выбирает контент, фиксирует порядок вариантов и создаёт все элементы дня.
5. Уникальное ограничение `(profileId, localDate)` защищает от двух параллельных созданий.
6. При конфликте повторный запрос читает уже созданную запись.

Случайный выбор должен быть воспроизводимым хотя бы внутри сохранённого дня. После записи результата повторно вычислять состав дня нельзя.

## Отправка ответа

Отправка первой попытки должна одной транзакцией:

1. проверить профиль и принадлежность элемента дню;
2. убедиться, что попытка ещё не зафиксирована;
3. вычислить результат на сервере;
4. сохранить исходный ответ, нормализованное значение и результат;
5. обновить соответствующую агрегированную статистику;
6. проверить завершение дня;
7. при первом завершении обновить streak;
8. вернуть новое состояние элемента и дня.

Повторная отправка не должна второй раз менять статистику. Для полного паритета её можно отклонять как уже завершённую попытку.

## Состояние и кэширование

- PostgreSQL — единственный источник истины для профилей и прогресса.
- `localStorage` хранит только выбор профиля и необязательные UI-настройки.
- Пользовательские страницы не должны получать общий cross-user cache.
- Статический учебный контент можно кэшировать, но персональный день и прогресс читаются свежими.
- После мутации интерфейс должен получить подтверждённое сервером состояние; optimistic UI допустим только с корректным rollback.

## UI-компоненты

Минимальный набор shadcn/ui:

- `Button`, `Card`, `Badge`;
- `Input`, `Textarea`, `Label`;
- `Progress`;
- `Tabs` или обычная навигация;
- `Alert` для обратной связи;
- `Dialog` для выбора профиля;
- `AlertDialog` для потенциально разрушительных действий;
- `Skeleton` для начальной загрузки профиля и дня.

Компоненты упражнений должны быть предметными (`GrammarExerciseCard`, `VocabularyCard`, `ConjugationCard`), а не копировать весь день в один client-компонент.

## Наблюдаемость

Для домашнего сервиса достаточно:

- структурированных логов в stdout контейнера;
- request/error context без полного текста пользовательских переводов;
- health endpoint;
- логирования ошибок миграции и подключения к БД;
- Docker healthchecks;
- понятных empty/error states в интерфейсе.

Метрики и внешний error tracking не обязательны для первой версии.
