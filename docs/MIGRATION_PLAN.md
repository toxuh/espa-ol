# План миграции и функциональный паритет

## Стратегия

Перенос выполняется вертикальными срезами, но HTML остаётся доступным для сравнения до конца. На первом проходе не переосмысливаются дизайн, учебная нагрузка и алгоритмы.

Главный принцип: сначала воспроизвести, затем улучшать.

## Этап 0. Зафиксировать эталон

- Сохранить исходный HTML неизменным.
- Добавить автоматизированный extractor или вручную подтверждённый экспорт всех банков контента.
- Создать unit-тесты на известные примеры нормализации, placement score, перевода и завершения дня.
- Зафиксировать количества контента из `CURRENT_PROTOTYPE.md`.
- Снять несколько эталонных сценариев поведения прототипа.

Готово, когда будущие изменения можно сравнивать не «на глаз», а по fixtures и тестам.

## Этап 1. Scaffold и инфраструктура разработки

- Создать Next.js App Router проект официальным `create-next-app` с TypeScript, Tailwind, ESLint и `src/`.
- Инициализировать shadcn/ui официальным CLI.
- Добавить Prisma и локальный PostgreSQL через Docker Compose.
- Ввести `.env.example`, health endpoint и базовые команды проверки.
- Добавить CI-подобную локальную команду: lint, typecheck, unit, integration, build.

На этом этапе интерфейс может быть пустой оболочкой; бизнес-функции ещё не заявляются готовыми.

## Этап 2. Контент и чистая доменная логика

- Перенести все банки контента в seed.
- Сохранить source IDs и порядок placement-вопросов.
- Реализовать и протестировать чистые функции:
  - `normalizeAnswer`;
  - `calculatePlacementLevel`;
  - `scoreTranslation`;
  - accuracy и adaptive priority;
  - allowed levels;
  - day completion;
  - streak transition.
- Проверить результаты на fixtures из HTML.

## Этап 3. Семейные профили

- Создать `Profile` и экран выбора имени.
- Сохранять `profileId` в `localStorage`.
- Показывать активный профиль в общей навигации.
- Реализовать смену профиля.
- Разделить все серверные запросы по `profileId`.
- Добавить предупреждение в документацию/UI, что это не защищённая авторизация.

## Этап 4. Placement и уровень

- Ручной выбор A1–C1.
- Тест из 15 вопросов с теми же вариантами и порогами.
- Сохранение результата и ответов.
- Повторный тест и ручная смена уровня без удаления истории.

## Этап 5. Генерация стабильного дня

- Создание `DailyPlan` по `(profileId, localDate)`.
- Атомарная генерация полного состава дня.
- Сохранение порядка MCQ-вариантов.
- Точное воспроизведение `±1` уровней, recent exclusions, accuracy priority и random noise.
- Идемпотентность при параллельных запросах.
- Проверка стабильности после reload и restart контейнера.

## Этап 6. Практика

Реализовать по отдельности:

1. Grammar MCQ.
2. Grammar fill-in.
3. Vocabulary.
4. Conjugation.
5. ES→RU.
6. RU→ES.
7. Reading.
8. Listening.

Для каждого блока обязательны:

- серверная проверка;
- фиксация первой попытки;
- правильная обратная связь;
- повторная загрузка сохранённого состояния;
- влияние на completion и статистику;
- E2E happy path и минимум один ошибочный ответ.

## Этап 7. Теория и адаптивное повторение

- Выбор слабого, нового или случайного урока в том же порядке приоритетов.
- Связанные упражнения.
- Общая попытка, если упражнение пересекается с практикой дня.
- Маркер нового урока или повтора.
- Порог повтора ниже 70%.

## Этап 8. Анализ, история и прогресс

- Промежуточный анализ дня.
- Итог дня.
- Сильные и слабые темы с правилами и примерами.
- История текущего и предыдущего месяцев.
- Детали выбранного дня.
- Точность тем с порогами `<45%`, `45–74%`, `≥75%`.
- Статистика слов и спряжений.
- Streak и число частичных/полных дней.

## Этап 9. Production-контейнеры и NAS

- Multi-stage Dockerfile.
- Production compose для app и db.
- Подключение app к существующей external-сети Traefik.
- PostgreSQL volume вне эфемерного слоя контейнера.
- Healthchecks и restart policy.
- Миграции перед запуском новой версии.
- Резервное копирование и тест восстановления.
- Проверка обновления без потери уже созданных дней.

## Матрица функционального паритета

Каждый пункт должен получить automated test или записанный manual acceptance result.

| Область          | Обязательная проверка                                         |
| ---------------- | ------------------------------------------------------------- |
| Профили          | два браузера/профиля не смешивают данные                      |
| Placement        | граничные scores 3/4, 6/7, 9/10, 12/13 дают правильные уровни |
| Ручной уровень   | A1–C1 сохраняются; история не удаляется                       |
| Дата             | один профиль получает один день в `Europe/Madrid`             |
| Reload           | состав, ответы и порядок вариантов сохраняются                |
| Grammar MCQ      | верный/неверный ответ и правило совпадают с HTML              |
| Grammar fill     | регистр, акценты и оговорённая пунктуация игнорируются        |
| First try        | повторный запрос не меняет статистику                         |
| Vocabulary       | все допустимые переводы принимаются                           |
| Conjugation      | карточка верна только при 6/6; маска полей сохраняется        |
| ES→RU            | reveal и три самооценки работают раздельно                    |
| RU→ES            | word-overlap и пороги 40/65 совпадают                         |
| Reading          | отметка сохраняется и участвует в completion                  |
| Listening        | каждый элемент отмечается отдельно                            |
| Theory           | слабая тема ниже 70% повторяется раньше новой                 |
| Adaptive grammar | слабая тема имеет больший базовый приоритет                   |
| Recent history   | лимиты 40/30/15 соблюдаются                                   |
| Completion       | день становится полным только после всех блоков               |
| Streak           | вчера + сегодня увеличивают серию; пропуск сбрасывает к 1     |
| Calendar         | partial и full отображаются различно                          |
| Analysis         | считаются только сделанные задания и первая попытка           |
| Restart          | перезапуск app/db не теряет прогресс                          |
| Backup           | восстановленная БД возвращает профили, дни и статистику       |

## Definition of Done всей миграции

- Все строки матрицы паритета проверены.
- Все исходные записи контента загружены и сверены по количеству и ID.
- Нет состояния учебного прогресса, которое существует только в React или `localStorage`.
- Два семейных профиля проверены end-to-end.
- Production build и контейнеры проходят healthcheck.
- Выполнено тестовое резервное копирование и восстановление PostgreSQL.
- Документы обновлены по фактической реализации.
- HTML перенесён из `temp` в явно обозначенный архив только после выполнения остальных пунктов.

## Улучшения после паритета

Отдельными решениями можно рассмотреть:

- уменьшение или настройку дневной нагрузки;
- повторную попытку после ошибки;
- строгий режим диакритики;
- более умную проверку перевода;
- встроенные задания к аудированию;
- редактирование контента;
- настоящую серверную сессию;
- PWA и offline mode;
- импорт локального состояния прототипа.
