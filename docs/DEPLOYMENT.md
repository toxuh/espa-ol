# Развёртывание на Synology DS923+

## Контекст

Целевой хост — домашний Synology DS923+, где уже работают другие Docker-проекты и Traefik. Español Diario не поднимает второй reverse proxy. По принятому решению приложение также публикуется на host-порту `8888` для прямого доступа из LAN; Traefik продолжает обращаться к внутреннему порту контейнера.

## Контейнеры проекта

Минимальная production-схема:

| Сервис | Назначение                | Публичный порт                                       |
| ------ | ------------------------- | ---------------------------------------------------- |
| `app`  | Next.js production server | `8888` на NAS; также доступен Traefik по Docker-сети |
| `db`   | PostgreSQL                | нет; доступен только внутренней сети compose         |

Одноразовый job `migrate` выполняет `prisma migrate deploy` и идемпотентный `prisma db seed` перед каждым запуском новой версии. `app` зависит от его успешного завершения. Development-команды и `prisma migrate dev` в production не используются.

## Сети

- `internal` — закрытая сеть между app и PostgreSQL.
- существующая external-сеть Traefik — подключается только к app.
- PostgreSQL не подключается к Traefik-сети.
- имя external-сети задаётся переменной или документируется при установке, поскольку оно зависит от текущей конфигурации NAS.

## Traefik

Compose проекта должен содержать labels для:

- включения маршрутизации только у app;
- правила `Host(...)` для выбранного домашнего домена;
- TLS через уже настроенный resolver, если он используется;
- внутреннего порта Next.js, обычно 3000;
- healthcheck path, например `/api/health`.

Точные domain, entrypoint, cert resolver и имя Docker-сети пока не фиксируются в репозитории и перечислены в `OPEN_DECISIONS.md`.

Host-порт задаётся переменной `APP_PORT` и по умолчанию равен `8888`. Внутри контейнера Next.js продолжает слушать `3000`, поэтому прямой адрес в локальной сети — `http://<nas-address>:8888`, а проверка состояния — `http://<nas-address>:8888/api/health`.

## Данные PostgreSQL

- Используется bind mount или named volume, размещённый на постоянном storage NAS.
- Путь не зашивается в общий compose: используется `.env` или NAS-specific override.
- Пароль БД хранится вне Git.
- `DATABASE_URL` доступен только app/migrate.
- База не публикуется в LAN без отдельной диагностической необходимости.

## Образ приложения

Рекомендуется multi-stage Dockerfile на Debian slim Node image. Это несколько тяжелее Alpine, но снижает количество проблем совместимости Prisma/OpenSSL на NAS.

Production image должен:

- содержать только runtime-артефакты;
- запускаться не от root;
- иметь `NODE_ENV=production`;
- использовать Next.js standalone output;
- не содержать `.env`, исходный HTML-прототип и development caches;
- поддерживать архитектуру `linux/amd64`, используемую DS923+.

## Переменные окружения

Минимальный ожидаемый набор:

```dotenv
DATABASE_URL=postgresql://...
POSTGRES_DB=espanol_diario
POSTGRES_USER=espanol_diario
POSTGRES_PASSWORD=...
APP_TIMEZONE=Europe/Madrid
NEXT_PUBLIC_APP_NAME=Español Diario
APP_PORT=8888
TRAEFIK_HOST=...
TRAEFIK_NETWORK=...
```

Переменные с `NEXT_PUBLIC_` не должны содержать секреты.

## Healthchecks

### PostgreSQL

Используется `pg_isready` с production-пользователем и базой.

### App

`GET /api/health` проверяет:

- что процесс Next.js отвечает;
- что выполняется простой запрос к PostgreSQL;
- опционально — что применена ожидаемая схема.

Endpoint не должен раскрывать DATABASE_URL, версии с известными уязвимостями или stack traces.

## Обновление

Безопасный порядок:

1. создать резервную копию БД;
2. скачать или собрать новый immutable image tag;
3. запустить compose: job `migrate` применит схему и синхронизирует контент;
4. дождаться запуска app после успешного job;
5. дождаться healthcheck;
6. проверить выбор профиля, текущий день и сохранённую историю;
7. сохранить предыдущий image tag для отката приложения.

Откат приложения не гарантирует откат схемы. Миграции должны быть backward-compatible там, где это возможно; разрушительные изменения требуют отдельного плана.

## Резервное копирование

Резервная копия считается рабочей только после тестового восстановления.

Минимальная политика для семейного проекта:

- ежедневный логический dump PostgreSQL;
- хранение нескольких последних ежедневных копий;
- дополнительная копия вне каталога активного volume;
- периодическое тестовое восстановление в отдельную базу;
- документированная команда восстановления.

Резервировать только Docker volume без согласованного snapshot недостаточно надёжно для работающего PostgreSQL.

## Эксплуатационные проверки

- NAS перезагружается, app и db автоматически поднимаются.
- Traefik видит только app.
- DB недоступна снаружи compose-сети.
- Переход через полночь `Europe/Madrid` создаёт новый день.
- Restart контейнера не создаёт второй DailyPlan.
- Одновременное открытие с двух устройств не дублирует попытки и статистику.
- Заполнение диска, недоступность БД и ошибка миграции дают понятный failure, а не бесконечную перезагрузку.
