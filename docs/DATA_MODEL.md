# Модель данных

## Принципы

- Учебный контент отделён от пользовательского прогресса.
- Стабильные строковые ID прототипа сохраняются как уникальные внешние ключи (`a1-01`, `v-a1-1`, `cj-a1-1` и так далее).
- День хранит снимок выбранного контента и порядка вариантов.
- Первая попытка неизменяема с точки зрения статистики.
- Исторические записи не должны ломаться после редактирования контента: критичные поля ответа и prompt при необходимости сохраняются snapshot-полями.
- Все пользовательские данные имеют обязательный `profileId` напрямую или через `dailyPlanId`.

## Основные сущности

### Profile

| Поле                | Назначение                                    |
| ------------------- | --------------------------------------------- |
| `id`                | UUID/CUID, значение хранится в браузере       |
| `name`              | отображаемое имя                              |
| `normalizedName`    | уникальное нормализованное имя                |
| `level`             | A1–C1 или `null` до placement                 |
| `placementDone`     | завершён ли стартовый выбор                   |
| `joinedAt`          | дата создания                                 |
| `timezone`          | обычно `Europe/Madrid`                        |
| `streak`            | текущая серия полных дней                     |
| `lastCompletedDate` | последняя локальная дата полного дня          |
| `active`            | позволяет скрыть профиль без удаления истории |

### GrammarExercise

- стабильный `sourceId`;
- уровень;
- topic;
- тип `MCQ` или `FILL`;
- prompt;
- варианты для MCQ;
- правильный ответ;
- HTML/текст объяснения;
- версия и признак активности.

### TheoryLesson

- стабильный `sourceId`;
- уровень;
- title и body;
- упорядоченная связь с grammar topics;
- версия и признак активности.

### ReadingContent

- стабильный `sourceId`;
- уровень;
- тип, title, text.

### ListeningContent

- стабильный `sourceId`;
- уровень;
- title, channel, URL.

### TranslationContent

- стабильный `sourceId`;
- направление `ES_RU` или `RU_ES`;
- уровень;
- title, source text, reference text.

### VocabularyCard

- стабильный `sourceId`;
- уровень;
- слово;
- основной перевод;
- допустимые ответы;
- контекст.

### ConjugationCard

- стабильный `sourceId`;
- уровень;
- инфинитив и перевод;
- внутреннее имя и label времени;
- шесть правильных форм;
- пример ES/RU.

### PlacementQuestion

- стабильный порядок;
- уровень вопроса;
- prompt, варианты, правильный ответ;
- признак активности.

## Пользовательский прогресс

### DailyPlan

Одна запись на профиль и локальную дату.

| Поле               | Назначение                           |
| ------------------ | ------------------------------------ |
| `id`               | идентификатор дня                    |
| `profileId`        | владелец                             |
| `localDate`        | дата без времени                     |
| `levelSnapshot`    | уровень на момент генерации          |
| `timezoneSnapshot` | временная зона генерации             |
| `theoryLessonId`   | урок дня                             |
| `theoryIsReview`   | новая тема или повтор                |
| `readingContentId` | чтение дня                           |
| `translateFromId`  | ES→RU                                |
| `translateToId`    | RU→ES                                |
| `completedAt`      | `null` или момент полного завершения |
| `createdAt`        | момент создания                      |

Уникальность: `(profileId, localDate)`.

### DailyGrammarItem

- `dailyPlanId`;
- `exerciseId`;
- секция `PRACTICE_MCQ`, `PRACTICE_FILL` или `THEORY`;
- позиция;
- сохранённый порядок вариантов;
- status;
- given answer;
- `correctFirstTry`;
- attemptedAt.

Один exercise может одновременно входить в практику и теорию текущего дня. Чтобы сохранить поведение прототипа, попытка должна быть общей для упражнения дня, а секции моделируются отдельными связями или массивом назначений, не дублирующими результат.

### DailyVocabularyItem

- позиция;
- status;
- given answer;
- correct first try;
- attemptedAt.

### DailyConjugationItem

- позиция;
- status;
- шесть введённых форм;
- маска правильности шести форм;
- результат всей карточки;
- attemptedAt.

### DailyListeningItem

- позиция;
- `completedAt` или `null`.

### DailyReadingProgress

Может быть полями в `DailyPlan`: `readingCompletedAt` и ссылка на контент.

### DailyTranslationResult

Для ES→RU:

- был ли открыт эталон;
- самооценка `EASY`, `MID`, `HARD`;
- timestamps.

Для RU→ES:

- введённый текст;
- процент совпадения;
- автоматически назначенная rating;
- checkedAt.

### PlacementAttempt

- профиль;
- время начала и завершения;
- ответы на 15 вопросов;
- общий score;
- назначенный уровень.

Ручной выбор уровня тоже должен оставлять событие смены уровня, даже если полноценная таблица аудита будет добавлена позже.

## Накопительная статистика

Для точного паритета допустимы материализованные таблицы:

- `ProfileTopicStat(profileId, topic, correct, total)`;
- `ProfileVocabularyStat(profileId, vocabularyCardId, correct, total)`;
- `ProfileConjugationStat(profileId, conjugationCardId, correct, total)`;
- `ProfileTheoryIntroduction(profileId, theoryLessonId, introducedAt)`.

Агрегаты обновляются в той же транзакции, что и первая попытка. Они ускоряют генерацию дня. Источником для восстановления остаются сохранённые попытки; позже должна появиться служебная команда пересчёта агрегатов.

## Недавние элементы

Отдельные массивы `recentGrammarIds`, `recentVocabIds` и `recentConjugationIds` не обязательны. Последние элементы можно получать запросом по `attemptedAt`:

- последние 40 грамматических попыток;
- последние 30 словарных попыток;
- последние 15 спряжений.

Это исключает рассинхронизацию массивов и истории.

## Целостность

- Все timestamps хранятся в UTC.
- `localDate` хранится отдельно и вычисляется в timezone профиля/приложения.
- Удаление контента, использованного в истории, запрещено; используется `active=false`.
- Удаление профиля в первой версии заменяется деактивацией.
- Числители статистики не могут превышать знаменатели.
- Одна первая попытка не может дважды увеличить агрегат.
- Полное завершение и streak обновляются идемпотентно.

## Seed и версионирование контента

Первый seed переносит данные из HTML один в один. Seed должен быть идемпотентным и обновлять контент по стабильному `sourceId`, не удаляя пользовательскую историю.

Рекомендуемый порядок:

1. извлечь банки прототипа в типизированные seed-модули;
2. проверить количество записей по типам и уровням;
3. загрузить через Prisma upsert;
4. выполнить integrity-тест ссылок theory topics и всех ответов;
5. запретить ручное дублирование тех же данных в UI-коде.
