# План развития учебного контента

Обновлено: 20 июля 2026 года.

## Назначение документа

Этот документ — handoff для отдельной сессии, посвящённой только учебному
контенту и инфраструктуре его seed-загрузки. Он фиксирует текущее состояние
банка, найденные дефекты, целевую организацию файлов и порядок расширения.

Главная цель — превратить механически перенесённый контент HTML-прототипа в
поддерживаемый учебный каталог, который можно безопасно вычитывать, расширять и
загружать в PostgreSQL без потери стабильности существующих учебных дней.

В рамках этой работы не меняются визуальный дизайн, дневная нагрузка,
адаптивный алгоритм и продуктовые правила первой попытки, если изменение не
нужно непосредственно для отображения более полного контента.

## Текущее состояние

В модульном каталоге `src/content/levels/<level>/<kind>.ts` находится 292
seed-элемента:

| ------------- | ----: | -------------------------- |
| Грамматика    |   102 | по 20 для A1–B2, 22 для C1 |
| Теория        |    30 | по 6                       |
| Словарь       |    70 | по 14                      |
| Спряжения     |    30 | по 6                       |
| Чтение        |    15 | по 3                       |
| Аудирование   |    10 | по 2                       |
| Перевод ES→RU |    10 | по 2                       |
| Перевод RU→ES |    10 | по 2                       |

Все 292 элемента имеют явные уникальные ID; 15 placement-ID были перенесены из
прежнего правила генерации по позиции и теперь зафиксированы в каталоге.

### Что уже является хорошей основой

- Уровни A1–C1 заполнены равномерно.
- Все грамматические упражнения имеют объяснение.
- MCQ и fill-in распределены почти поровну.
- Теоретические уроки не являются заглушками: в них есть правила, формы и
  примеры; текущий объём — примерно 114–168 слов на урок.
- Длина чтения и переводов постепенно увеличивается от A1 к C1.
- Стабильные ID сохранены из прототипа.
- Seed идемпотентно обновляет записи по `sourceId` и деактивирует отсутствующий
  контент вместо физического удаления.
- Банк уже покрывает полный пользовательский цикл MVP.

Это хороший parity-базис, но не достаточный каталог для длительного обучения.

## Найденные проблемы

### P0. Ошибки и нарушения целостности

#### Неправильные канонические ответы

Шесть ответов хранятся без обязательного графического ударения:

| ID    | Сейчас           | Должно быть      |
| ----- | ---------------- | ---------------- |
| a1-12 | `lapices`        | `lápices`        |
| a2-01 | `compre`         | `compré`         |
| b1-04 | `hablaria`       | `hablaría`       |
| b2-09 | `habia empezado` | `había empezado` |
| b2-13 | `queria`         | `quería`         |
| b2-14 | `cual`           | `cuál`           |

Нормализация ответа должна по-прежнему принимать ввод без диакритики, но
канонический ответ, объяснение и показанный пользователю эталон обязаны быть
написаны правильно.

#### Слово не выделяется в контексте

Компонент ищет точное вхождение `word`, но в семи контекстах используется
изменённая форма:

- `v-b2-1`: exigir → exigen;
- `v-b2-3`: desempeñar → desempeña;
- `v-b2-4`: abarcar → abarca;
- `v-b2-5`: plantear → plantea;
- `v-b2-7`: carecer → carece;
- `v-b2-8`: suscitar → suscitó;
- `v-c1-7`: arraigado → arraigada.

Нужно добавить явное поле `contextTarget` и валидировать, что именно оно
присутствует в `context`. Поле `word` остаётся словарной формой.

#### Другие точечные дефекты

- `b2-05` неоднозначен: в текущем предложении допустимы как `Estudiando
mucho`, так и `Al estudiar mucho`.
- В `th-a2-6` ошибочно написано «сравнение
  превосходства/уступительности».
- Объяснение `ser/estar` через абсолютное «постоянное/временное» слишком
  категорично. Его нужно оставить как начальную подсказку, но дополнить более
  точной моделью: идентификация/характеристика против состояния/местоположения,
  с контрастными примерами и исключениями.
- Автоматический пример в теории заменяет только первый `___` и некорректен для
  заданий с несколькими пропусками. Для карточки нужен явный `solvedExample`.

### P1. Короткая ротация

Текущий банк исчерпывается слишком быстро относительно дневной нагрузки:

- чтение повторяется после трёх дней на уровне;
- каждый вид перевода — после двух дней;
- теория — после шести уроков;
- на уровне есть ровно два listening-элемента, поэтому почти один и тот же
  набор показывается каждый день;
- доступных спряжений меньше дневного лимита 20, поэтому генератор практически
  ежедневно возвращает весь допустимый пул;
- грамматика на краевых уровнях начинает заметно повторяться через 2–3 дня.

Адаптивное повторение является частью продукта, но оно должно возвращать
слабый материал осознанно, а не потому, что нового материала больше нет.

### P1. Недостаточная учебная глубина

#### Грамматика

Текущее `explain` обычно содержит одно правило и один пример. Для полноценной
обратной связи нужны:

- короткая подсказка до ответа;
- объяснение выбора формы в конкретном предложении;
- контраст с вероятной ошибкой;
- дополнительные примеры;
- типичная ошибка русскоязычного ученика;
- явная связь с уроком теории;
- отдельные допустимые ответы и один канонический ответ.

#### Теория

Каждый урок должен иметь одинаковую педагогическую структуру:

1. цель урока;
2. краткая интуиция;
3. полное правило или схема;
4. таблица форм, если применимо;
5. 4–6 примеров с переводом;
6. контрастные примеры;
7. типичные ошибки;
8. краткая шпаргалка;
9. ссылки на упражнения закрепления.

Сейчас восемь упражнений по шести темам не связаны ни с одним уроком:

- `b1-10`, `b1-13`, `b1-14`;
- `b2-13`;
- `c1-09`, `c1-11`, `c1-17`, `c1-19`.

#### Словарь

Нужно добавить часть речи, род существительных, значимые формы, управление,
сочетания, форму слова в контексте, перевод контекста и заметку о регистре или
употреблении.

#### Спряжения

Нужны правило образования, неправильная основа или изменение корня, временные
маркеры, минимум два примера и типичная ошибка. Шесть форм и один пример
остаются проверяемым ядром карточки.

#### Чтение

К каждому тексту нужны:

- краткое задание перед чтением;
- словарь из 4–10 выражений;
- 3–5 вопросов на понимание;
- ответы и короткое объяснение;
- при необходимости культурная или языковая заметка.

#### Переводы

Нужны несколько допустимых эталонов, ключевые конструкции, разбор сложных мест
и примечания о естественном порядке слов. Текущий word-overlap остаётся
приблизительной оценкой до отдельного продуктового решения.

#### Аудирование

Сейчас карточки ведут преимущественно на каналы или большие плейлисты. В десяти
карточках только семь уникальных URL, а некоторые ссылки повторяются между
уровнями. Для каждого элемента нужны:

- ссылка на конкретное видео или эпизод;
- длительность;
- вариант испанского и примерная скорость речи;
- инструкция, что именно слушать;
- 3–5 вопросов;
- transcript, полный или учебный фрагмент, если это разрешено источником;
- дата последней проверки ссылки.

#### Placement

Три вопроса на уровень недостаточны для устойчивой диагностики. Вопросам нужны
явные ID, `topic`, объяснение и отметка проверяемого навыка. Расширение банка и
изменение алгоритма выбора 15 вопросов должны выполняться отдельным
согласованным срезом, чтобы не изменить пороги незаметно.

### P1. Нет настоящей валидации seed-данных — закрыто этапом 1

Каталог теперь проходит Zod/runtime-валидацию до обращения к БД.
`npm run content:check` проверяет дополнительные межэлементные инварианты,
минимальные объёмы и manifest защищённых смысловых полей.

### P1. Изменение контента может переписать смысл старого дня

`DailyPlan` хранит source IDs, а при открытии приложение загружает актуальный
`ContentItem.data`. Если под существующим ID изменить prompt, ответ или смысл,
старый день покажет новую карточку рядом со старой попыткой.

Правило до реализации полноценной revision-модели:

- орфография, пунктуация и расширение объяснения без изменения проверяемого
  смысла могут сохранять ID;
- изменение prompt, правильного ответа, типа задания или проверяемого правила
  требует нового ID и деактивации старой карточки;
- `content:check` должен уметь обнаруживать изменение защищённых полей по
  сохранённому manifest/digest;
- позднее можно заменить это правило на явную ревизию в БД и ссылку дневного
  плана на конкретную ревизию.

## Целевая организация файлов

Прежний `prototype-content.json` имел около 2 750 строк и удалён из
канонического пути после проверенного структурного переноса.

Рекомендуемая структура:

```text
src/content/
  schemas.ts
  catalog.ts
  shared/
    levels.ts
    placement.ts
  levels/
    a1/
      grammar.ts
      theory.ts
      vocabulary.ts
      conjugations.ts
      readings.ts
      translations.ts
      listening.ts
    a2/
      ...
    b1/
      ...
    b2/
      ...
    c1/
      ...
```

Правила организации:

- основной размер дробления — один уровень и один вид контента;
- не создавать отдельный файл на каждое упражнение или слово;
- теорию делить по отдельным урокам только тогда, когда файл уровня становится
  действительно неудобным;
- `catalog.ts` должен явно импортировать и объединять все модули;
- seed импортирует проверенный каталог, а не читает монолитный JSON;
- `scripts/extract-prototype-content.mjs` остаётся архивным одноразовым
  инструментом, но больше не генерирует канонический каталог;
- порядок элементов должен быть детерминированным и не зависеть от glob или
  порядка обхода файловой системы.

TypeScript-модули предпочтительнее ручных JSON-файлов: они дают `satisfies`,
многострочные строки, комментарии и раннюю проверку типов. Zod всё равно нужен
для полной runtime-валидации перед seed.

## Целевые контракты контента

Названия полей можно уточнить при реализации, но обязательные возможности
фиксируются сейчас.

### GrammarExercise

- `id`, `revision`, `level`, `topic`, `lessonIds`;
- `type`, `prompt`, `options`;
- `answer`, `acceptedAnswers`;
- `hint`;
- `explanation`;
- `solvedExample`;
- `examples[]`;
- `commonMistakes[]`.

### TheoryLesson

- `id`, `revision`, `level`, `title`, `topics`;
- `objectives[]`;
- структурированные `sections[]` вместо произвольной HTML-строки;
- `summary[]`;
- `examples[]` с испанским текстом и переводом;
- `commonMistakes[]`;
- `exerciseIds[]` или проверяемая обратная связь через topic registry.

### VocabularyCard

- `id`, `revision`, `level`;
- `word`, `partOfSpeech`, `gender`, `forms`;
- `translation`, `acceptedTranslations`;
- `context`, `contextTarget`, `contextTranslation`;
- `collocations[]`, `usageNote`, `register`.

### ConjugationCard

- текущие verb/tense/forms;
- `formationRule`, `irregularity`, `markers[]`;
- `examples[]` вместо одного примера;
- `commonMistakes[]`.

### ReadingContent

- текущие title/text;
- `instructions`, `glossary[]`, `questions[]`, `notes[]`;
- ответы на вопросы внутри trusted seed-контента.

### TranslationContent

- текущие text/reference;
- `alternativeReferences[]`;
- `keyConstructions[]`;
- `notes[]`;
- для RU→ES — список допустимых ключевых слов или выражений отдельно от полного
  эталона.

### ListeningContent

- текущие title/channel/url;
- `durationMinutes`, `dialect`, `speed`, `instructions`;
- `questions[]`, `transcript` или `summary`;
- `lastVerifiedAt`.

### PlacementQuestion

- явный `id`, `level`, `topic`, `skill`;
- текущие prompt/options/answer;
- `explanation`;
- позднее — данные калибровки, если появится достаточно попыток.

## Обязательные проверки

`content:check` должен завершаться ошибкой при любом из условий:

- повторяющийся ID;
- неизвестный уровень или topic;
- пустое обязательное поле;
- неправильное число форм спряжения;
- MCQ-ответ отсутствует среди вариантов после нормализации;
- обязательный `contextTarget` отсутствует в контексте;
- упражнение с несколькими пропусками не имеет явного `solvedExample`;
- theory topic не имеет упражнений;
- упражнение с обязательной theory-связью осталось orphan;
- listening URL невалиден;
- защищённые смысловые поля существующего ID изменены без новой ревизии;
- каталог не достигает зафиксированного минимального объёма.

После каждого законченного среза выполняются:

```bash
npm run content:check
npm run db:seed
npm run verify
npm run docker:config
```

Seed нужно запускать дважды и подтверждать одинаковый результат. Для изменений
рендера требуется browser acceptance всех затронутых типов карточек.

## Критерий завершения инициативы

- Каталог разделён по уровню и типу, монолит больше не редактируется вручную.
- Все элементы проходят runtime-валидацию до обращения к БД.
- P0-дефекты исправлены.
- Каждый тип контента имеет полноценный учебный шаблон.
- A1–C1 наполнены до согласованных ориентиров либо имеют документированное
  исключение.
- Адаптивное повторение вызывается прогрессом пользователя, а не постоянной
  нехваткой материалов.
- Старые попытки и дни не получают несовместимый новый prompt/answer.
- Документы соответствуют реальному формату каталога и seed-процессу.
