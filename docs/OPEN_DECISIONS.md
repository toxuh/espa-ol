# Открытые решения

Ни один из этих пунктов не блокирует документирование проекта. Перед затрагивающим его implementation-срезом решение нужно перенести в соответствующий основной документ.

## Решено в MVP

- **OD-001:** любой домашний пользователь может создать уникальный профиль по имени. Это удобство, не публичная регистрация.
- **OD-002:** персональные экраны — client bootstrap; `profileId` передаётся в `X-Profile-Id`. Cookie пока не нужна.
- **OD-004:** полная исходная дневная нагрузка сохранена.
- **OD-005:** после первой проверки карточка заблокирована; статистика неизменяема.
- **OD-006:** перевод проверяется исходным word-overlap с порогами 40/65.
- **OD-009:** PostgreSQL зафиксирован на `17-bookworm`.
- **OD-010:** HTML поступает только из доверенного seed; пользовательский HTML не рендерится.

## Остаётся решить

### OD-001 (архив). Управление семейными профилями

Варианты:

- заранее seed-ить фиксированный список имён;
- разрешить любому посетителю создать профиль вводом имени;
- сделать скрытую служебную страницу управления профилями.

Позже можно добавить скрытое переименование/деактивацию профилей. На текущий учебный цикл это не влияет.

### OD-002 (архив). Серверный bootstrap профиля

`localStorage` недоступен Server Components при первом SSR. Возможны:

- client-side bootstrap и внутренние fetch-запросы с `X-Profile-Id`;
- синхронизация выбранного ID из `localStorage` в cookie;
- полностью клиентская оболочка персональных экранов.

Если персональные страницы позднее переводятся в Server Components, можно синхронизировать ID в несекретную cookie. Это не будет авторизацией.

### OD-003. Импорт прогресса из HTML

Нужно решить, существует ли уже ценный `espanolDiario_v4` state в семейных браузерах.

- Если нет — начинаем с чистых профилей.
- Если да — перед миграцией нужен одноразовый импорт JSON из `localStorage` с валидацией и привязкой к профилю.

Не следует реализовывать импорт заранее без реальных данных.

### OD-004 (после паритета). Дневная нагрузка

Текущий день очень велик: 20 грамматических заданий, 10 слов, до 20 спряжений, два перевода, чтение, аудирование и теория. Для паритета это сохраняется.

После миграции можно отдельно решить:

- сделать короткий и полный режим;
- уменьшить количество спряжений;
- считать streak по минимальному обязательному набору;
- позволить пользователю настроить нагрузку.

### OD-005 (после паритета). Повторная попытка

Прототип блокирует карточку после первой проверки. Можно позднее разделить:

- неизменяемый `correctFirstTry` для статистики;
- возможность исправить ответ для обучения;
- отдельный статус `masteredAfterRetry`.

До отдельного решения UI сохраняет блокировку прототипа.

### OD-006 (после паритета). Проверка перевода

Word-overlap сохраняется для паритета, но не является качественной оценкой языка. Будущие варианты:

- несколько допустимых эталонов;
- stemming/lemmatization;
- rubric и ручная самооценка;
- локальная или внешняя LLM-проверка.

Для семейного self-hosted проекта нельзя добавлять внешний AI-сервис без отдельного решения о стоимости и приватности.

### OD-007. Production-параметры Traefik

Перед деплоем нужно узнать и зафиксировать:

- hostname;
- имя external Docker network;
- entrypoint;
- TLS resolver;
- нужно ли ограничение только LAN/VPN;
- формат labels, уже используемый соседними проектами.

Эти значения следует снять с реально работающего compose одного из существующих проектов на NAS, а не придумывать.

### OD-008. Backup retention

Нужно выбрать фактический каталог, расписание и срок хранения dump-файлов с учётом существующей backup-системы Synology.

### OD-009 (архив). Версия PostgreSQL и стратегия образов

Версия фиксируется при scaffold после проверки совместимости актуальных Prisma и NAS. Major-обновление PostgreSQL не должно происходить автоматически через плавающий Docker tag.

### OD-010 (архив). HTML объяснений теории

Прототип хранит explanation/body как доверенный HTML. При переносе возможны:

- sanitised HTML;
- Markdown;
- типизированные rich-text blocks.

Для быстрого паритета допустим sanitised HTML из seed, но прямой `dangerouslySetInnerHTML` с редактируемым пользовательским вводом запрещён.
